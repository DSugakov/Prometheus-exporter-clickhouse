# Требования: ClickHouse Prometheus Exporter

## Тема

Разработка продвинутого Prometheus-exporter для ClickHouse без необходимости изменения server-side конфигурации ClickHouse.

## Технологический стек

- **Язык реализации:** Go (Golang).
- **Клиент ClickHouse:** `github.com/ClickHouse/clickhouse-go/v2` (native TCP; при необходимости HTTP можно добавить отдельным транспортом).
- **Метрики Prometheus:** `github.com/prometheus/client_golang`.
- **Конфигурация:** YAML + переменные окружения с префиксом `CH_EXPORTER_`.

## Цель

Создать внешний сервис с эндпоинтом `/metrics`, который:

- разворачивается **без доступа** к конфигурационным файлам ClickHouse на сервере;
- собирает **существенно больше** метрик, чем минимальные решения, с **контролем кардинальности**;
- поддерживает **профили сборов:** `safe`, `extended`, `aggressive`;
- **минимально нагружает** ClickHouse (таймауты, лимиты конкурентности, лимиты строк, отключение тяжёлых коллекторов в `safe`).

## Цель и границы

- Подключение только как **клиент** (учётная запись, TLS, DSN). Правки `config.xml` / встроенного Prometheus-endpoint на стороне CH **не требуются**.
- Экспортёр не заменяет аудит PII: в лейблы **не** попадают пользовательские строки запросов и персональные данные.

## Подключение к ClickHouse

- **Транспорт:** native TCP (порт по умолчанию 9000) через clickhouse-go.
- **TLS:** опционально; поддержка кастомного CA (файл) для корпоративных сред.
- **Поведение при сбоях:** таймауты на запрос; частичный сбой коллектора не обязан ронять весь ответ `/metrics` (ошибки отражаются в метриках `ch_exporter_collector_*`).

## Развёртывание и модель наблюдаемости

- Рекомендуемый вариант: **один процесс экспортёра на ноду ClickHouse** (sidecar / агент), scrape Prometheus с лейблом `instance` целевого хоста CH.
- Допустим централизованный вариант (один экспортёр → несколько CH), если явно задано в конфиге; увеличивается нагрузка на сеть и CH.

## Безопасность

- Отдельная учётная запись с **минимальными** правами: `GRANT` только на чтение нужных `system.*` таблиц (см. `docks/grants_example.sql` при необходимости).
- Пароль и DSN: через env или файл; **не** логировать пароль и полный DSN.

## Совместимость версий ClickHouse

- **Минимальная целевая версия:** 23.x (LTS-ветка; при необходимости уточняется по стенду).
- Недоступная таблица или колонка: коллектор пропускается или упрощается (**feature detection**), процесс не завершается.

## Нефункциональные требования

- Время ответа `/metrics` ограничивается конфигом (`scrape_timeout` / таймаут коллекторов).
- Ограничение параллельных запросов к CH на один scrape.

## Профили сборов

| Профиль    | Содержание (коллекторы) |
|-----------|-------------------------|
| `safe`    | `system.metrics`, `system.events`, `system.asynchronous_metrics` |
| `extended`| + агрегаты по репликации, merges/mutations, диски, краткая сводка по parts |
| `aggressive` | + больше детализации там, где допустимо без взрыва cardinality (настраиваемые лимиты) |

## Задачи (трассировка к ВКР)

- Проанализировать источники метрик и зафиксировать baseline для экспериментов (см. `docs/metrics_matrix.md`).
- Реализовать экспортёр и артефакты внедрения (Docker, CI, дашборд Grafana).

## Критерии готовности

- Бинарник/образ без требования server-side конфига CH.
- Профили и контроль cardinality.
- Сравнение с baseline по методике из `docs/experiments.md`.
- CI: сборка, тесты, линтер.
